# eda MCP — workspace orientation

A short reference covering everything you need to know about the project
layout, file-write rules, encoding conventions, and the mechanical bits
the tool descriptions don't have room for. Fetch via
`resources/read { uri: "eda://docs/workspace" }`.

## Project layout

```
projects/<project>/
├── src/                 designs (each .sexp = one design-block)
├── lib/
│   ├── components/      part definitions ((component …) / (component-family …))
│   ├── modules/         reusable sub-circuits ((defmodule …))
│   ├── pinouts/         pin id → function name maps (auto-generated, see below)
│   ├── footprints/      PCB land patterns
│   ├── datasheets/      PDF datasheets (binary, read-only from MCP)
│   ├── models/          3D STEP models (read-only from MCP)
│   ├── parts/           curated part data (read-only from MCP)
│   └── sources/         original .kicad_sym / .kicad_mod files (writable)
├── blocks/              shared block diagrams
├── history/             auto-snapshots (read-only; use restore_version)
├── reviews/             saved review state
└── out/                 build output (regenerated; not writable)
```

### What's writable from MCP

| Path                 | read | write | notes                                        |
| -------------------- | ---- | ----- | -------------------------------------------- |
| `src/`               | ✅   | ✅    | designs and per-design `.bom`                |
| `lib/components/`    | ✅   | ✅    |                                              |
| `lib/modules/`       | ✅   | ✅    |                                              |
| `lib/pinouts/`       | ✅   | ✅    | usually auto-generated; see Pinouts below    |
| `lib/footprints/`    | ✅   | ✅    |                                              |
| `lib/sources/`       | ✅   | ✅    | KiCad sources behind pinouts/footprints      |
| `lib/datasheets/`    | ✅   | ❌    | PDFs added by hand on disk or via browser UI |
| `lib/models/`        | ✅   | ❌    | imported by humans                           |
| `lib/parts/`         | ✅   | ❌    | imported by humans                           |
| `blocks/`             | ✅   | ✅    |                                              |
| `history/`            | ✅   | ❌    | use `restore_version` to revert              |
| `out/`                | ✅   | ❌    | regenerated by `build`                       |

`auth/`, `oauth_*.json`, `.env`, `.key`, `.pem` and similar are denied
unconditionally. `.layout` and `.bom` are regenerated by `build` and not
directly writable.

If `write_file` returns `permission denied`, check the response's `hint`
field — the server tries to point you at the right tool when one exists
(e.g. `history/` → `restore_version`, `lib/datasheets/` → drop the PDF on disk).

## File response shapes

### Mutations (`write_file`, `edit_file`, `delete_file`, `move_file`)

Both `dirty_designs` and `library_changes` ride on every response:

```json
{
  "path": "lib/components/foo.sexp",
  "bytes_written": 1234,
  "sha256": "...",
  "dirty_designs": [],
  "library_changes": [
    { "kind": "component", "name": "foo", "path": "lib/components/foo.sexp" }
  ]
}
```

- `dirty_designs` lists designs whose `(import …)` graph reaches the
  modified file. Empty when nothing imports the file yet — that's fine,
  it just means there's nothing to rebuild.
- `library_changes` is the unambiguous "the write landed at a library
  entity named X" signal. Empty for paths outside `lib/`.

### `read_file`

Text files come back in `content`, binary files (`.pdf`, `.png`,
`.step`, `.zip`, etc.) in `base64`, with a `binary: true|false` flag.
Always returns `sha256` of the full file so a follow-up write can
pass `expected_sha256` for CAS.

## Pinouts: source of truth and editing them

`lib/pinouts/<name>.sexp` carries the header
`;; Auto-generated pinout — DO NOT EDIT`. The source of truth is the
matching `lib/sources/<name>.kicad_sym` file. To change a pin name:

1. `describe_component { name: "<part>" }` — the response's `pinout_source`
   field tells you which `.kicad_sym` produced the pinout.
2. Edit that file in `lib/sources/` (it's writable).
3. Call `regenerate_pinout { source: "<that path>" }`.

The pinout is regenerated and the old contents are overwritten.

Names get mangled in the upstream KiCad symbol surprisingly often
(`VPOS5` rendered as `VPOS`, `RFIN4+` rendered as `REFIN4+`). Don't
work around it in the component file — fix it in the source. After
the first `regenerate_pinout` runs, the pinout includes a
`;; source: <path>` header that `describe_component` reads back.

Component-side requirements reference pins by their **function name**
from the pinout (`(pin "VDD")`), not by physical pin id. The function
name is the `"…"` string in `(pin <id> "<function>")`.

## `(id <hex>)` tags on instances

8-character lowercase hex (must start with a-f so it parses as an atom,
not a number). Example: `(id ab12cd34)`.

- Optional at write-time. The build inserts one for every instance that
  doesn't have one and writes the source back to disk — so after one
  build every instance has a stable id.
- Used as primary keys for: schematic SVG element identity, PCB
  placement persistence (`<name>.layout`), sub-block instance
  derivation across reused modules.
- NOT exported to KiCad — purely an internal stable handle.

You can omit ids entirely on new instances; the build will mint them.
You can copy an existing one to keep PCB placement during a refactor.

## String literals and Unicode

The tokenizer is byte-pass-through inside `"…"`. Unicode characters
work verbatim — `µ`, `°`, `±`, em dashes, anything UTF-8. No escape
syntax is needed for non-ASCII. Atoms (the un-quoted symbol form) are
ASCII-only; if you need any other character, use a quoted string.

Inside quoted strings the only escapes the parser understands are
those produced by JSON's standard string encoding when the MCP
transport delivers them (`\"`, `\\`, `\n`, `\r`, `\t`, `\uXXXX`).
You don't need to manually escape Unicode.

## Common workflows

**Build a schematic from scratch**
1. `list_library` → see what components exist.
2. `describe_component { name: "<part>" }` for each part you want to
   instantiate — gives pinout and library requirements in one call.
3. `write_file src/<design>.sexp` → the design body.
4. `build { name: "<design>" }` → version bumps; live viewers refresh.

**Add a requirement to an existing component**
1. `describe_component` to inspect the current state.
2. `edit_file lib/components/<name>.sexp` to add the `(requirement …)`
   form. Response will include
   `library_changes: [{ kind: "component", name: "…" }]` even when no
   design imports the part yet.
3. `build` any design that does import it (`dirty_designs` lists them).

**Fix a typo in an auto-generated pinout**
1. `edit_file lib/sources/<name>.kicad_sym` to fix the function name.
2. `regenerate_pinout { source: "lib/sources/<name>.kicad_sym" }`.
3. `build` any consumer designs.

**Add a datasheet PDF**
1. Ask the user to drop the PDF in `projects/<project>/lib/datasheets/`
   (or upload via the browser's library page) — there's no MCP write
   path for PDFs.
2. Reference the saved filename from a component via
   `(datasheet "<name>.pdf")` and from requirements via
   `(ref "<name>.pdf" (page N))`.
