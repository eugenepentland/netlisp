- [ ] 2026-05-15 (b4056072) BOOT0 is hardwired to GND via R_BOOT0 (10k pulldown). This forces flash boot and blocks USB DFU / UART boot entry, including first-boot HSLV provisioning. Replace with a 3-pin header or momentary pushbutton from BOOT0 to 3V3, keeping the 10k as a default pulldown. Without this fix the board cannot enter DFU on a virgin chip without probing TP7 high during reset.
- [ ] 2026-05-15 (7edbe959) BOOT1 (PA6) is unassigned and will float at reset. Per UM3234 §3.2.7 the boot ROM samples BOOT1 *before* BOOT0, so a floating PA6 makes the latched BOOTSR[1] non-deterministic and can drop the board into dev-boot or serial-boot at random. Tie PA6 to GND through a 10k pulldown (or assign it to a deliberate static-low signal).
- [ ] 2026-05-15 (e5e05fee) PH1 is unassigned. Per UM3234 §3.6.4.2 the boot ROM samples PH1 twice to autodetect HSE crystal vs analog-bypass vs digital-bypass. With a crystal (Y1 ABM8) on OSC_IN/OSC_OUT, you want oscillator mode — meaning PH1 levels must read as "not both-low and not both-high". A floating PH1 can sample either way and cause occasional cold-boot failures. Either tie PH1 to a deterministic mixed state per AN5967, or plan to blow OTP_WORD16.bit2 (disable_hse_bypass_detect=1) during provisioning.
- [ ] 2026-05-15 (92f1e636) No fallback serial-boot recovery path. None of USART1 (PE5/PE6), USART2 (PA2/PF6), or UART4 (PA0/PA1) are exposed. If USB-C is damaged or a buggy FSBL kills DFU, the only recovery is SWD over the off-board 6-pin header — not a usable field-service path. Bring UART4 (PA0/PA1) out to two unused expansion pins or a 4-pin debug header (TX/RX/GND/3V3). The boot ROM scans all UART RX pins in parallel, so no firmware cost.
- [ ] 2026-05-15 (34cc8fe1) PG10 (UART5_TX boot-failure trace pin) is not exposed. Per UM3234 §3.3.2 and §3.10, when the boot ROM hits a blocking failure it dumps a 64-bit status word (Table 20) over UART5_TX at 9600 baud — invaluable for diagnosing "chip won't boot" silent failures. Currently PG10 floats unused inside the package. At minimum add a test point; ideally route to the expansion connector or a UART debug header alongside the UART4 fallback.
- [ ] 2026-05-15 (da858faf) HSLV provisioning sequence needs to be documented. VDDIO3 is correctly tied to V1P8 (1.8V) for the MX66UW flash on Port N, but a virgin STM32N6 ships with OTP124[15] (HSLV) unblown, meaning the Port N pads default to 3.3V drive at first boot. The conservative boot SPI mode (1-1-1, 0xB Fast Read, Table 14) usually still works at reduced margin, but ST's documented sequence is to blow HSLV before relying on it. Plan an EOL provisioning step: enter USB DFU (requires the BOOT0 fix above), set OTP124[15] and PWR_SVMCRx VDDIOxVRSEL, reset, verify flash boot.
- [ ] 2026-05-15 (206f801d) Rename section "XSPI2 NOR Flash" → "Boot NOR Flash (XSPIM_P2 / Port N)". The flash is physically on PN0–PN11, which is XSPIM_P2 — exactly where the boot ROM expects it (UM3234 Table 28). The current name suggests the flash is on the wrong controller and will trip up the next reviewer (and future-you). Cosmetic but worth fixing now while it's fresh.
- [ ] 2026-05-15 (cc18574d) Add a boot-fail LED on PG10. Per UM3234 §3.3.2 and §3.10, on a blocking boot failure the boot ROM drives PG10 (BOOTFAILN, also UART5_TX) low open-drain and holds it there. Wire an LED+resistor from VDD to PG10 so it lights solid-on whenever the boot ROM gives up — instant eyeball-level "did the chip die at boot" indicator without needing a UART receiver. Pair with note 34cc8fe1 (test point/header on PG10 for the actual 9600 baud status trace); the LED and the trace tap can share the net.

- [ ] 2026-05-15 (dce126da) BNO08x pins 15 and 16 need 10k pullups to VDD. The BNO08x boot ROM samples these as part of an I2C-presence/protocol-select check at startup; without the pullups it may latch the wrong interface mode and the SPI link won't come up. Add two 10k resistors to VDD inside the bno08x-imu module on pins 15 and 16.
- [ ] 2026-05-15 (efccc79b) AD7380 VLOGIC (pin 2) is currently tied to V1P8 in the ad7380-channel module — should be tied to VDD (3.3V) instead, on all three ADC instances. The schematic comment in the ADC Array section already documents the intended behavior ("VLOGIC (pin 2) = 3.3V (tied to VDD) ... matches the STM32's VDD-bank GPIOs without level shifting"), but the module implementation has it on V1P8. Fix the module so VLOGIC → VDD. At 1.8V VLOGIC the SDO outputs would be 1.8V swing into a VDD-bank GPIO, marginal but not guaranteed; on VDD the digital interface is clean 3.3V end-to-end.

- [ ] 2026-05-15 (66349b0c) USB-C connector shield tabs aren't soldering down — the shield-tab through-holes in the footprint are unplated mechanical holes instead of plated vias, so the shell legs pass through but make no electrical/mechanical bond to the board. Fix the usb-c-hs module footprint so those holes are plated through-holes (PTH pads) tied to GND. Without this the connector is held on by only the signal pads and will rip off the board on the first cable yank — and there's no chassis-ground path for ESD/EMI either.
- [ ] 2026-05-16 (6c8d880d) Add a test point for the 2.5V reference rail (VREF_2V5 from the LTC6655). The Test Points section currently covers VBATT, VDD, V1P8, VDDCORE, NRST, PG_3V3, BOOT0, and PWR_ON, but not VREF_2V5 — which is the rail most worth probing during ADC bring-up.
